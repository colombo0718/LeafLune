# 全葳的核心價值與必要經驗

> 關鍵觀念：  
> **沒有在真實公司把「會員＋金流＋通訊＋舊系統」玩熟，LeafLune 的 AI 會員中樞就很難長得健全。**  
> 全葳就是你現在可以用來練這一整套的「實戰實驗室」。

---

## 0. 為什麼是全葳，而不是別家公司？

- 做的是 **POS / ERP / 進銷存 / 會員**，天生就是「資料庫＋金流＋會員」的集中地。
- 老闆本來就想往 **進階行銷系統、AI 小助手、會員經營** 走，方向和你一致。
- 你現在已經能碰到：
  - 客戶資料庫
  - 各種報價 / 庫存 / 對帳邏輯
  - LINE OA、通用任務 Bot、SQL Gateway…
- 對你來說，本質上是：  
  > **老闆花薪水讓你練「AI 會員資料庫系統」這門必修課。**

---

## 1. 全葳的核心價值（對你 & 對 LeafLune）

### 1.1 真實商用級「資料庫 × 工作流」戰場

- 各種實際存在的：
  - 客戶資料表、商品表、庫存表、訂單表、點數表…
  - VB / C# / SQL 混在一起的舊程式碼與報表。
- 這些都是未來 LeafLune 要處理的「會員＋行為＋金流」縮影，只是 domain 換成教育。

> 在這裡搞懂「舊系統是怎樣長出來的」，未來你在 LL 才不會造出一個一樣難維護的怪獸。

---

### 1.2 完整又「髒」的客戶資料庫

- 全葳有：
  - 多年累積的客戶 & 終端店家。
  - 訂單、維修、點數、簡訊、帳單紀錄。
- 這些資料：
  - **不乾淨、不統一、欄位歷史包袱多** —— 正是訓練你「資料治理」的最佳場域。
- 如果你能在這種環境下建立：
  - 清楚的 **Schema 理解、欄位對照、視圖 / 匯出層**，
  - 未來要在 LeafLune 上設計自己的 DB，就會非常有底氣。

---

### 1.3 金流與帳務完整鏈條

- 全葳實際面對：
  - 客戶怎麼付錢（轉帳 / 刷卡 / 第三方金流）。
  - 公司怎麼開發票、對帳、退款、補單。
- 你可以學到：
  - 付款成功的「技術流程」 + 「會計 / 業務流程」怎麼對得上。
  - 金流 webhook 進來後，資料怎麼寫進 DB，怎麼產生「可查詢的訂單歷史」。

> 未來 LeafLune 要賣的是課程 / 課表 / 競技門票，  
> 但底層的「訂單 → 權限」邏輯，跟全葳現在做的事情非常像。

---

### 1.4 LINE OA / 通訊與會員的串接

- 全葳想做的：
  - LINE OA 綁定客戶。
  - 用 OA 做客服 / 發通知 / 串報價或查詢。
- 你在這裡可以練：
  - LINE OA webhook 的整個生命週期。
  - LIFF 或外部網頁怎麼帶 token、怎麼反查 DB 。
  - 通用任務 Chatbot 工作流：  
    精準抽出意圖 → 查 DB / 呼叫 API → 把結果丟回 LINE。

> 這一整套 之後可以 1:1 移植到 LeafLune：  
> 由 LINE / Zalo 當入口，帶會員進入 RR / XX / SS 平台。

---

### 1.5 老闆願意投資「AI＋行銷系統」

- 你不是在一間完全排斥 AI 的公司裡偷做 side project。
- 老闆本來就想推：
  - 進階行銷系統
  - 自動報價 / 簡訊 / 推播
  - AI 小幫手輔助客服與業務
- 代表：
  - 你可以光明正大地用「公司需求」來試驗你要的 AI 架構，
  - 做出來的東西 **全葳直接受惠**，  
    而 **你也把這套架構學熟帶回 LeafLune**。

---

## 2. 你在全葳必須磨練到的能力模組

> 可以把它當成「全葳修煉 Checklist」，每一點都是之後 LeafLune Core 的基石。

### 2.1 資料庫與舊系統的「讀懂力」

- 能夠：
  - 看一段 VB / SQL / 報表，就知道它在用哪些表、哪些欄位。
  - 走訪整個「報價 → 出貨 → 對帳」流程，畫出資料流圖。
- 具體成果：
  - 為現有系統畫出  
    **「資料模型圖（ERD）」+「主要業務流程圖」**。
  - 能對每個重要欄位解釋：「這是什麼？哪裡會用到？可以改嗎？」

---

### 2.2 AI × 舊系統的整合實戰

- 練習把：
  - 通用任務 Chatbot  
  - SQL Gateway  
  - 任務管理 / 報價小幫手
- 真正掛上全葳的 DB、業務流程，而不是玩 demo。

你要學會的關鍵點：

1. **如何讓 LLM 只做「解讀」和「決策」，不直接碰 DB。**  
2. 把最終動作交還給：
   - 明確定義的 API（例如 `generateQuote`, `queryPoints`…）
   - 或可審核的 SQL / 程式碼。
3. 把這整套流程「文件化」，方便未來換模型 / 換供應商。

---

### 2.3 會員 / 客戶資料治理與分群

- 在全葳練習：
  - 怎麼用既有欄位（消費金額、來店頻率、產品類型…）分群。
  - 如何設計額外欄位或 table 來記錄「行為歷程」（例如使用某功能次數）。
- 練習產出：
  - 「客戶分群」報表。
  - 對應的行銷策略（例如：某群客戶適合被推哪種方案）。

> 之後 LeafLune 把「教育內容、學習任務、競技賽季」換上來，  
> 同一套分群邏輯就可以直接用。

---

### 2.4 金流 / 訂單 / 權限的端到端理解

你在全葳要搞懂：

1. 客戶怎麼決定買什麼？（報價 → 成交）  
2. 錢實際怎麼流？（金流、匯款、對帳、發票）  
3. 系統怎麼記錄與查詢這些歷史？  
4. 如果要「退費 / 補單 / 改方案」，資料怎麼變？

> 這會直接變成你在 LeafLune 設計  
> `orders` / `order_items` / `entitlements` 的「直覺」。

---

### 2.5 通訊整合：LINE OA / Webhook / LIFF

你在全葳至少要實作過一個完整流程：

1. 使用者在 LINE OA 送出訊息或按選單。  
2. Webhook 收到 → 判斷意圖 → 呼叫公司 API / DB。  
3. 回傳結果（文字、Flex Message、圖文選單…）。  
4. 視情況觸發：
   - 開啟 LIFF 頁面（進入 Web App）
   - 寫入任務 / log / 工單

這套經驗以後可以直套在：

- 司簿星（MM 任務管理）
- RR 平台（查訓練紀錄、發任務）
- XX（提醒教案、提醒作業）
- SS（發賽季結果、對戰邀請）

---

### 2.6 規格、文件化與版本控制

在全葳你要刻意練習：

- 每做一個新功能，一定會留下：
  - 資料結構說明（欄位表 / ERD）
  - API 規格（input / output / 範例）
  - 使用流程（哪一種使用者會在什麼情境下用到）
- 並且：
  - 這些東西最好放在 Git / Notion / Wiki，  
    用「版本」管理，而不是散在 Chat / PDF 裡。

> 這會讓未來的 LeafLune 生態不會變成「只有創作者自己看得懂的混亂宇宙」。

---

### 2.7 商業視角與溝通（老闆、RD、客服、業務）

- 在全葳，你可以練：
  - 老闆要的是「營收 / 新客 / 維繫舊客」，  
    你要學會把 AI 功能翻譯成這些語言。
  - 和 RD / MIS 討論「風險、維護成本、架構」。
  - 和 業務 / 客服 討論「實際好不好用」「會不會多一堆工作」。

> 之後 LeafLune 的合作對象會變成：  
> 校方、補教、家長、學生、出版社。  
> 在全葳練會的「翻譯與溝通能力」，到那邊一樣適用。

---

### 2.8 資安與權限意識

在全葳這邊是你學習：

- 哪些資料不可亂暴露給外部服務（包含 LLM）。  
- API / webhook 一定要做：
  - 身分驗證
  - 權限控管
  - 記錄 log 以便 audit
- 哪些操作需要：
  - 雙重確認（例如重建 DB、批次調整庫存）
  - 或只能在後台做，不能讓 Bot 自己亂跑

> 這些都是未來 LeafLune 成為一個「真正可以收費、承擔責任的服務」必備的底層素養。

## 4. 建議的在全葳修煉路線圖

> 這不是給公司的 OKR，是給你自己的「武功心法」。

### 4.1 第 0 ～ 3 個月：全面認識系統

- 把：
  - 客戶資料表、訂單表、庫存表、點數 / 簡訊相關表  
    梳理成一份簡單 ERD。
- 每個主要功能（報價、出貨、對帳）畫出：
  - **使用者流程**（誰在什麼畫面做什麼）  
  - **資料流程**（寫進 / 讀出哪些表）

---

### 4.2 第 3 ～ 6 個月：做出「AI×資料庫×LINE」的完整範例

例子（可依實際需求換）：

- 「AI 報價小助手」或「AI 查庫存小幫手」：
  1. LINE OA 或 Web 輸入自然語言。
  2. 通用任務 Bot 判斷意圖 → 呼叫 API / SQL Gateway。
  3. 回覆結果給使用者。
  4. 同時記錄一筆 log / 任務在內部系統。

做到這樣，你就有一個可以 8 成照搬到 LeafLune 的 blueprint。

---

### 4.3 第 6 ～ 12 個月：會員與行銷系統的雛形

- 在全葳試著：
  - 把客戶分群（高價值、潛在流失、新客…）。
  - 為每一類設計不同的訊息節奏、優惠或提醒。
- 技術面：
  - 熟悉固定排程（每日 / 每週自動跑腳本）。  
  - 熟悉匯出報表給業務 / 行銷使用。

這一段練好，以後 LeafLune 的「學習路徑推薦 / 任務推送」就是換個 domain 而已。

---

### 4.4 一年後：把「AI 會員資料中樞」變成你的招牌技能

- 你在全葳的定位，變成：
  - 不是只會寫幾個腳本的工程師，
  - 而是「懂業務 + 懂資料庫 + 懂 AI + 懂金流 + 懂會員經營」的系統設計者。
- 同時：
  - LeafLune Core 的設計、RR 平台、司簿星 MM、各品牌會員系統  
    都是吃同一套腦袋長出來的。

---

> 總結一句：  
> **全葳 = 你練「AI 會員資料庫 × 金流 × 通訊 × 舊系統整合」的實戰道場。**  
> 你在這裡打磨出的所有技能，  
> 未來都會直接成為 LeafLune 宇宙的骨架與護城河。
